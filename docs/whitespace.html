Whitespace steganography, explained from zero

The trailing space/tab method (SNOW-style). We hide an encrypted secret in the invisible whitespace at the end of text lines. This page walks one tiny payload through every single step, including what it looks like on paper and exactly where it breaks.

1. The one-sentence idea

A byte is 8 bits. Each bit is either 0 or 1. A line of text can quietly carry one bit by ending in an ordinary space (means 0) or a tab character (means 1). Stack enough lines and you can hide any message as invisible trailing whitespace — no strange characters, no colors, nothing the eye notices.

2. The cast of characters

ThingWhat it isRole in the trick
secretthe message you actually want to sendencrypted first, then hidden
passworda string only you and the recipient knowturns the secret into unreadable ciphertext
coverinnocent-looking text everyone seescamouflage; payload is added to its line ends
stego textthe cover with hidden bits appendedwhat you actually send
space ␣ASCII 0x20, the normal spaceencodes a 0 bit
tab \tASCII 0x09encodes a 1 bit

3. The full pipeline (end to end)

  1. Encrypt. The secret's UTF-8 bytes go through Fernet (AES-128-CBC + HMAC-SHA256) keyed by SHA-256(password). Out comes ciphertext — random-looking bytes. Even if someone later finds the hidden bits, they still need the password.
  2. Length-prefix. Prepend the ciphertext length as a 4-byte big-endian integer. This is the header that lets the decoder know when to stop reading bits.
  3. Bits. Convert [4-byte length] + ciphertext to one long binary string. Every byte becomes 8 characters of 0/1.
  4. Append. Split the cover by \n. For each bit in order, take the next line and append a ␣ (bit 0) or \t (bit 1) to its end. If the cover runs out of lines, pad with empty lines (each empty line still gets one trailing space/tab — see Limitations).
  5. Send. The receiver gets text that looks normal. They do not need any special software beyond something that preserves trailing whitespace.
  6. Decode. For every line, walk backwards from the end: space→0, tab→1, stop at the first non-whitespace char. Concatenate the bits.
  7. Read header. First 32 bits = length in bytes. Read that many more bytes as the ciphertext.
  8. Decrypt. Feed ciphertext + password back through Fernet. Out comes the original secret. Wrong password → Fernet's HMAC rejects it, you get nothing.

4. The worked example, on paper

Let's hide the 1-byte secret 'A' (= 0x41) in this 3-line cover using password "pw". We'll do every step by hand so you can see the bits.

Step A — encrypt

encrypt_message(b"A", "pw") asks Fernet for ciphertext. Fernet adds a random IV, so the real bytes are unpredictable; for this walkthrough pretend the ciphertext comes out to exactly one byte: 

ciphertext = b"\x41"      # 1 byte, value 0100 0001

In real use Fernet's output is much longer (it prepends a version byte, timestamp, IV, and appends an HMAC). The math is identical — just more bits. We use 1 byte here so the page fits.

Step B — length-prefix

The length of the ciphertext is 1. Encode 1 as a 4-byte big-endian integer, then concatenate the ciphertext:

length (4 bytes):  00 00 00 01
ciphertext (1 byte): 41
payload (5 bytes):  00 00 00 01 41

Step C — to bits

Each byte → 8 bits. 5 bytes → 40 bits total:

00       00       00       01       41
00000000 00000000 00000000 00000001 01000001

So the full bit string we need to hide is:

00000000 00000000 00000000 00000001 01000001
└────── header (32 bits) ────────┘└ payload ┘

Step D — append to cover lines

Cover is 3 lines, payload is 40 bits → we need 40 lines, so the encoder pads with 37 empty lines. Bit 0 → space, bit 1 → tab. The first few lines become:

hello world␣ # bit 0 → trailing space it is nice␣ # bit 0 → trailing space today␣ # bit 0 → trailing space ␣ # bit 0 (padded empty line) ␣ # bit 0 (padded empty line) ... ...more zeros... →| # bit 33 → the lone '1' (tab), this is the "00000001" byte ␣ # bit 34 ␣ # bit 35 ␣ # bit 36 ␣ # bit 37 →| # bit 38 → '1' (the '01000001' byte's high bit) ␣ # bit 39 → '0'

Notice the giveaway: the cover had 3 short lines, but the stego text now has 40 lines, most of them empty. That is the single biggest detection signal for this method and you cannot avoid it — see §6.

Step E — decode (reverse)

For each line, read backwards: tab=1, space=0, stop at non-whitespace. You get the same 40-bit string back:

00000000 00000000 00000000 00000001 01000001

First 32 bits → int(...,2) = 1, so the decoder knows to read 1 more byte. The next 8 bits → 01000001 = 0x41 = b"A". Hand b"\x41" and "pw" to Fernet → original secret.

5. Doing it yourself on paper

  1. Pick a short secret and a short cover (a couple of sentences, one idea per line).
  2. Write the secret's bytes in binary (an ASCII table is fine; for arbitrary bytes use Python: ' '.join(f'{b:08b}' for b in secret.encode())).
  3. Count the bits. Add 32 for the length header. That number = how many lines your cover needs. If your cover is shorter, the encoder invents blank lines — visible to a careful reader.
  4. For each bit, append ␣ (0) or \t (1) to the next line. Use a text editor that shows whitespace (VS Code: toggle "Render Whitespace").
  5. To decode: for each line strip the trailing whitespace, recording 0 for space and 1 for tab. First 32 bits → length, next length×8 bits → ciphertext bytes.

You can skip the paper and just use the demo or the CLI: echo "cover" | python steg.py encode -m ws -p pw -s "A" -o stego.txt then python steg.py decode -m ws -p pw -i stego.txt.

6. Where it fails

Whitespace stripping. Most Markdown renderers, code formatters (Prettier, Black with --trim-trailing-whitespace), git diff with -w, and many chat apps delete trailing whitespace on save or render. If even one trailing space/tab is removed, the bit string shifts and decode fails (returns b"").

Line reflow. Re-wrapping text (word processors, Gmail compose) joins and splits lines. The bit-per-line assumption breaks. The payload is gone.

Line-count blowup. Each payload bit eats one line. A 1 KB ciphertext needs (4 + 1024) × 8 = 8224 lines. A short cover gets padded with thousands of empty lines — instantly suspicious.

Existing tabs in the cover. A \t already inside a line would be misread as a 1-bit by the decoder (it walks backwards from the end and tabs are tabs from either side). The encoder refuses covers that contain any tab. Same for pre-existing trailing whitespace — the encoder rejects it to avoid ambiguity.

HTML / rich text. Web pages collapse runs of whitespace and ignore trailing whitespace entirely. Pasting rendered HTML destroys the payload. Only transport as plain text.

7. Is it safe?

Two separate questions. People conflate them and that's where the mistakes happen.

Can the hidden message be read?

Yes, against the secret's contents: it is encrypted. Fernet is AES-128-CBC with HMAC-SHA256 authentication. The key is SHA-256(password). Even with the stego text and the password, the attacker still needs the password to decrypt. Wrong password → HMAC mismatch → Fernet throws InvalidToken, nothing leaks.

No, against the key. Key derivation is plain SHA-256 — no salt, no PBKDF2. An attacker who has the ciphertext can brute-force a weak password offline at full hash speed. The README marks this as a known simplification; if your threat model includes offline brute force, switch to PBKDF2HMAC with a random salt (and ship the salt — it can be public).

Is the existence of the message hidden?

No. Steganography hides content from humans, not from analysis. python steg.py detect -m ws -i text.txt counts trailing tabs; many lines ending in tabs is positive proof a payload exists. The detect command even reads the 4-byte length header and tells the attacker exactly how many bytes are hidden, no password required. The password protects the contents, not the existence.

8. Limits (short version)

  • Capacity. 1 bit per line. Headers and HMAC overhead mean even a 100-character secret needs a cover of hundreds of lines.
  • Lossy transport is deadly. Anything that strips, rewraps, or HTML-renderes trailing whitespace kills the payload. Screenshots, OCR, TTS, paste-into-Word: all gone.
  • Channel is suspicious. A normal person's text file does not have trailing tabs on half its lines. Detection is easy.
  • No integrity without the password. A bit flip in transit corrupts the ciphertext; Fernet's HMAC catches it but you lose the whole message — no partial recovery.
  • Cover must be tab-free and trailing-whitespace-free. The encoder rejects covers that violate this rather than silently mis-encoding.

9. When to pick this method

Pick whitespace when the channel preserves trailing whitespace but strips zero-width characters — e.g. raw text files, plain-text email, source-code comments, git diff without -w. For everything else (HTML, chat apps, rendered Markdown) the zero-width method usually survives better.
