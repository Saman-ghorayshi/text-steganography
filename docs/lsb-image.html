Image (PNG LSB) steganography, explained from zero

The least-significant-bit method. We hide an encrypted secret in the least-significant bit of each R, G, B channel of a PNG cover. This page walks one tiny payload through every single step, shows you exactly what the pixels look like, and exactly where it breaks.

1. The one-sentence idea

A pixel's color is stored as three numbers — red, green, blue — each one byte (0–255). Changing the lowest bit of any one of those numbers changes the color by at most 1 unit out of 255, which the human eye cannot see. So every pixel can quietly carry three payload bits (one per channel), and the picture still looks identical.

2. The cast of characters

ThingWhat it isRole in the trick
secretthe message you actually want to sendencrypted first, then hidden
passworda string only you and the recipient knowturns the secret into unreadable ciphertext
coveran innocent-looking PNG everyone seescamouflage; payload is written into its pixel LSBs
stego imagethe cover with the payload stored in LSBswhat you actually send

LSBleast-significant bit, the rightmost binary digitwhat we flip; we don't touch the other 7 bits of any channel

3. The full pipeline (end to end)

  1. Encrypt. The secret's UTF-8 bytes go through Fernet (AES-128-CBC + HMAC-SHA256) keyed by SHA-256(password). Out comes ciphertext — random-looking bytes. Even if someone later finds the hidden bits, they still need the password.
  2. Length-prefix. Prepend the ciphertext length as a 4-byte big-endian integer. This is the header that lets the decoder know when to stop reading bits.
  3. Bits. Convert [4-byte length] + ciphertext to one long binary string. Every byte becomes 8 characters of 0/1.
  4. Walk pixels. Open the cover PNG, read pixels row by row. For each pixel we have three channel bytes (R, G, B) (we skip alpha if present, so transparency stays identical). For each payload bit in order, take the next channel and overwrite its LSB with the bit's value.
  5. Save as PNG. Re-encode as PNG (lossless). Any other format would re-quantize and destroy the LSB payload — see §6.
  6. Send. The receiver gets an image that looks identical. They do not need any special software beyond software that preserves the PNG bytes exactly.
  7. Decode. Read pixels row by row, channel by channel, concatenate every LSB into one long bit string.
  8. Read header. First 32 bits = length in bytes. Read that many more bytes as the ciphertext.
  9. Decrypt. Feed ciphertext + password back through Fernet. Out comes the original secret. Wrong password → Fernet's HMAC rejects it, you get nothing.

4. The worked example, on paper

Let's hide the 1-byte secret 'A' (= 0x41) in a 2×2 PNG cover using password "pw". We'll do every step by hand so you can see the bits. (A 2×2 cover has 12 channel bytes, so it can hold 12 bits — too small in practice; we use it so the page fits.)

Step A — encrypt

encrypt_message(b"A", "pw") asks Fernet for ciphertext. Fernet adds a random IV, so the real bytes are unpredictable; for this walkthrough pretend the ciphertext comes out to exactly one byte: 

ciphertext = b"\x41"      # 1 byte, value 0100 0001

In real use Fernet's output is much longer (it prepends a version byte, timestamp, IV, and appends an HMAC). The math is identical — just more bits.

Step B — length-prefix

The length of the ciphertext is 1. Encode 1 as a 4-byte big-endian integer, then concatenate the ciphertext:

length (4 bytes):  00 00 00 01
ciphertext (1 byte): 41
payload (5 bytes):  00 00 00 01 41

Step C — to bits

Each byte → 8 bits. 5 bytes → 40 bits total. 40 bits need 40 channel bytes, so the 2×2 cover we use here would in reality not be big enough — pretend we have a 4×4 cover (48 channels, plenty).

00       00       00       01       41
00000000 00000000 00000000 00000001 01000001

So the full bit string we need to hide is:

00000000 00000000 00000000 00000001 01000001
└────── header (32 bits) ────────┘└ payload ┘

Step D — write into pixel LSBs

Assume the first ~16 pixels of the cover (in row-by-row order) start with these four, and we ignore the rest for clarity:

pixel 0:  (R=64,  G=128, B=192)   binary: 01000000 10000000 11000000
pixel 1:  (R=12,  G=200, B=33)     binary: 00001100 11001000 00100001
pixel 2:  (R=120, G=15,  B=80)     binary: 01111000 00001111 01010000
pixel 3:  (R=255, G=0,   B=128)    binary: 11111111 00000000 10000000
...

We walk channels in order R, G, B, then next pixel. The first 8 bits of the length header are 00000000, so the first 8 channel LSBs all become 0. Bits 8–15 are also 00000000 → next 8 channel LSBs become 0, bits 16–23 (also all zeros) → next 8 channel LSBs become 0. Then the last byte of the length header is 00000001, so the 32nd LSB we touch becomes 1 (the only 1 in the header). Then the 8 payload bits (the byte 0x41 = 01000001):

bit 32 = 0   → channel LSB becomes 0
bit 33 = 1   → channel LSB becomes 1
bit 34 = 0   → channel LSB becomes 0
bit 35 = 0   → channel LSB becomes 0
bit 36 = 0   → channel LSB becomes 0
bit 37 = 0   → channel LSB becomes 0
bit 38 = 0   → channel LSB becomes 0
bit 39 = 1   → channel LSB becomes 1

In total we touch 40 channel bytes — that's 13 full pixels (39 channels) plus one more channel. Channel numbering starts at 0: bit 0 → pixel 0 R, bit 1 → pixel 0 G, bit 2 → pixel 0 B, bit 3 → pixel 1 R, etc. So:

bit 31 (last bit of the length header = 1) → pixel 10, B channel
bit 32 (first payload bit = 0)              → pixel 11, R channel
bit 33 (payload bit = 1)                    → pixel 11, G channel
...
bit 39 (last payload bit = 1)               → pixel 13, R channel

Listing only the channels whose LSB actually flips:

pixel 10, B = 01000000 → 01000001  (header: last bit of "00000001")
pixel 11, R = 11111111 → 11111110  (payload bit 32 = 0: cleared)
pixel 11, G = 00000000 → 00000001  (payload bit 33 = 1: set)
pixel 13, R = 10000000 → 10000001  (payload bit 39 = 1: set)

Notice the giveaway: a real cover PNG (let alone a real-world photograph) has LSBs that are roughly 50/50 ones and zeros — they're mostly noise. Our modified cover has its first ~32 LSBs almost entirely 0 because the length header is small. A steganalysis tool can detect this: see §7.

Step E — decode (reverse)

Read pixels row by row, channel by channel. Concatenate each LSB. You get the same 40-bit string back:

00000000 00000000 00000000 00000001 01000001

First 32 bits → int(...,2) = 1, so the decoder knows to read 1 more byte. The next 8 bits → 01000001 = 0x41 = b"A". Hand b"\x41" and "pw" to Fernet → original secret.

5. Doing it yourself

  1. Open the demo (the demo tab doesn't yet cover image LSB — only the Python CLI does for now). To run the CLI: python steg.py encode -m img -p pw -s "A" -c cover.png -o stego.png then python steg.py decode -m img -p pw -i stego.png.
  2. Pick a small PNG cover that is bigger than 16×16 or so — capacity is width × height × 3 / 8 − 4 bytes.
  3. The encoder refuses non-PNG output (JPEG, WebP, etc.) because lossy re-encode would silently destroy the LSB payload.
  4. To decode: open the stego PNG, walk pixels and channels, collect LSBs, reassemble bytes.

For a programmatic look the Python library is the canonical source: from steg import hide_img, reveal_img — takes PNG bytes, returns PNG bytes / secret bytes.

6. Where it fails

Lossy re-encode. Re-saving the stego PNG as JPEG, WebP, or any lossy codec quantizes the DCT coefficients. The LSB bits you so carefully wrote into pixel channels get shifted around and the payload is destroyed — silently. The CLI refuses .jpg/.webp output for this method, but if you re-encode by hand the tool can't help you.

Re-quantize or rebuild. Some image tools rebuild pixels when you edit them — recompress, downscale, change color profiles. Anything that doesn't preserve the exact byte stream of the PNG pixels will destroy the payload.

Cover too small. Capacity is width × height × 3 / 8 − 4 bytes of ciphertext. A 32×32 PNG holds only (32×32×3 → 3072 bits) / 8 − 4 = 380 bytes minus the Fernet overhead (~57 bytes for an empty payload). For real secrets pick at least 256×256.

Statistical steganalysis. A natural photo's LSBs look like random noise — roughly 50/50 ones and zeros. A payload-bearing region of a PNG has its first bytes filled with the length header, which is highly structured (lots of leading zero bits for typical small payloads). A steganalyzer with a chi-square estimator or a structural-anomaly detector can flag this. Our detect helper uses the simplest honest signal — a readable length header that fits inside the LSB capacity is itself suspicious — but is not a robust steganalyzer.

Alpha channel is skipped. We don't write into alpha because that would change transparency visibly. If you have a cover that is already fully transparent, no payload is hidden there, and capacity is smaller than for a fully-opaque image (actually it's the same — we still walk R, G, B regardless — but not using the alpha plane is a missed capacity option for a future revision).

7. Is it safe?

Two separate questions. People conflate them and that's where the mistakes happen.

Can the hidden message be read?

Yes, against the secret's contents: it is encrypted. Fernet is AES-128-CBC with HMAC-SHA256 authentication. The key is SHA-256(password). Even with the stego image and the password, the attacker still needs the password to decrypt. Wrong password → HMAC mismatch → Fernet throws InvalidToken, nothing leaks.

No, against the key. Key derivation is plain SHA-256 — no salt, no PBKDF2. An attacker who has the ciphertext can brute-force a weak password offline at full hash speed. Same limitation as the text methods; the README marks this as a known simplification.

Is the existence of the message hidden?

No. Steganography hides content from humans, not from analysis. python steg.py detect -m img -i stego.png reads the LSB stream and reports the distribution of 1s vs 0s. It also reads the 4-byte length header and tells the attacker exactly how many bytes are hidden, no password required. A real steganalysis goes further with chi-square estimators and structural LSB-pair histograms; our detect is the honest minimum length-header sanity check, suitable for a teaching tool — a production-quality image steganalyzer is named in the project plans as a polish follow-up.

8. Limits (short version)

  • Capacity. 3 bits per pixel. A 1024×1024 PNG hides 1024×1024×3/8 − 4 ≈ 384 KiB. Plenty for text secrets; tiny for media.
  • Lossy transport is deadly. Any encoder that re-quantizes (JPEG, lossy WebP) destroys the payload — silently. Stick to PNG.
  • Steganalysis is well-studied. LSB in the spatial domain is the textbook steg method and the textbook steganalyzer target. A determined analyst will find your payload — pick a different cover if you need genuine secrecy.
  • No integrity without the password. A bit flip in transit corrupts the ciphertext; Fernet's HMAC catches it but you lose the whole message — no partial recovery.
  • Skipped planes. Alpha is untouched for invisibility — but that means capacity is 75% of what's available. Higher-bit-plane writes are out of scope because they'd change pixel values enough to be visible.

9. When to pick this method

Pick the image method when the cover you control is a PNG and the transport channel preserves bytes — e.g. posting a PNG on a site that doesn't re-encode, sending one as a file attachment, embedding in a ZIP. For text channels, whitespace or zero-width work without inventing an image. The image method has the highest capacity of the three and is the smallest diff to the existing encrypt/length-prefix pipeline — at the cost of "only as lossless as you keep it."
