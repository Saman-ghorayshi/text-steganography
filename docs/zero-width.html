Zero-width Unicode steganography, explained from zero

We hide an encrypted secret in invisible Unicode characters inserted into the cover text. The text looks identical to the eye. This page walks one tiny payload through every single step, including what it looks like on paper and exactly where it breaks.

1. The one-sentence idea

Some Unicode characters render with zero width â€” they exist in the string but take no visible space. We use two of them as binary digits: U+200B (zero-width space) means 0, U+200C (zero-width non-joiner) means 1. Insert enough of them into ordinary text and you can hide any message without changing how the text looks.

2. The cast of characters

You will sometimes see a third character, U+200D (zero-width joiner) mentioned. We deliberately do not use it â€” two characters are enough for binary and the decoder stays simpler. U+200D in cover text is ignored on decode, not stripped.

3. The full pipeline (end to end)

  1. Encrypt. The secret's UTF-8 bytes go through Fernet (AES-128-CBC + HMAC-SHA256) keyed by SHA-256(password). Out comes random-looking ciphertext. Even if someone finds the hidden bits, the contents are still protected by the password.
  2. Length-prefix. Prepend the ciphertext length as a 4-byte big-endian integer. This is the header the decoder needs to know where the payload ends.
  3. Bits. Convert [4-byte length] + ciphertext to one binary string. Every byte â†’ 8 chars of 0/1.
  4. Insert. Map each bit to U+200B (0) or U+200C (1) and insert the whole block right after the first character of the cover. (The decoder ignores position â€” it reads every zero-width char in the text. Placement is cosmetic. See Limitations.)
  5. Send. The receiver gets text that displays byte-for-byte like the cover. To the eye it is unchanged.
  6. Decode. Scan the entire text. For each char that is U+200B or U+200C, append 0 or 1 to a bit string. Skip every other character.
  7. Read header. First 32 bits = length in bytes. Read that many more bytes as the ciphertext.
  8. Decrypt. Feed ciphertext + password to Fernet. Wrong password â†’ HMAC rejects, nothing leaks.

5. Doing it yourself on paper

  1. Pick a short secret and any cover (one word is enough â€” the cover barely matters for capacity here).
  2. Write the secret's bytes in binary. For arbitrary bytes: ' '.join(f'{b:08b}' for b in secret.encode()).
  3. Prepend 32 zero-bits (placeholder for the length â€” you'll fill in the real length at the end). Total bits needed = 32 + 8 Ã— (number of ciphertext bytes).
  4. For each bit write U+200B (0) or U+200C (1). Type them with \u200b and \u200c in Python, or copy from a Unicode picker.
  5. Insert the whole block after the first character of your cover. (Anywhere works â€” decode ignores position. Placement only matters for steganalysis resistance.)
  6. To decode: programmatically count/filter the two char classes in the text. Doing this by hand is impractical, which is the point â€” the eye can't see the payload.

Skip the paper and use the demo or CLI: echo "hello" | python steg.py encode -m zw -p pw -s "A" -o stego.txt then python steg.py decode -m zw -p pw -i stego.txt.

6. Where it fails

Sanitizing platforms. WhatsApp, Discord, Telegram in some modes, GitLab rendered Markdown, and most rich-text email clients strip U+200B/U+200C on paste or render. The payload is silently deleted. Test your channel before trusting it.

Encoding round-trips. If text gets re-encoded through a system that drops "format" or "invisible" Unicode categories (some legacy pipelines, ASCII-only transports), the characters vanish. Any byte lost means the length header no longer matches and decode returns b"".

The cluster is detectable. This tool inserts all the invisible chars as one block right after the first character. A steganalyst scanning the byte offsets of U+200B/U+200C sees a giant spike at position 1 â€” proof of a payload and a hint it's this tool. Distributing the chars evenly across the cover would help; the code marks this as a known simplification. See zw_encode in steg.py.

Cover validators occasionally reject zero-width chars. Some forms (passwords, usernames, payment fields) treat zero-width chars as suspicious and reject the whole string. Don't try to hide a payload inside a username.

Lossy visual transport. Screenshots, OCR, text-to-speech, and printed paper all drop the invisible chars. The payload only survives digital copy-paste that preserves Unicode.

7. Is it safe?

Two separate questions. People conflate them and that's where the mistakes happen.

Can the hidden message be read?

Against the secret's contents: yes, it is encrypted. Fernet is AES-128-CBC + HMAC-SHA256. The key is SHA-256(password). Even given the stego text and the password, the attacker still needs the password to decrypt. Wrong password â†’ HMAC mismatch â†’ Fernet throws InvalidToken, nothing leaks.

Against the key: no. Key derivation is plain SHA-256 â€” no salt, no PBKDF2. An attacker with the ciphertext can brute-force a weak password offline at full hash speed. Marked as a known simplification in the README; upgrade to PBKDF2HMAC + random salt if offline brute-force is part of the threat model.

Is the existence of the message hidden?

No â€” and worse than the whitespace method here. U+200B and U+200C essentially never appear in normal prose. Any count above zero is positive evidence of a payload. python steg.py detect -m zw -i text.txt counts the invisible chars, divides by 8 for the hidden byte count, and reads the 4-byte length header to say exactly how many bytes are hidden â€” all without the password. Steganography hides content from humans, not from analysis tools.

8. Limits (short version)

  • Capacity. Practically unlimited â€” cover length barely matters because the invisible chars take no visible space. But the more you insert, the bigger the cluster and the more detectable the payload becomes.
  • Sensitive to transport. Any platform that sanitizes invisible Unicode kills the payload. There is no graceful degradation â€” one stripped char means the length header is wrong means decode fails entirely.
  • Trivially detectable. A character class count is enough. Zero resistance to statistical steganalysis.
  • Clustered placement. All payload chars sit after the cover's first character in this implementation. Better resistance would distribute them; the decoder doesn't care where they live so this is a one-function change.
  • No integrity without the password. A bit flip in transit corrupts the ciphertext; Fernet's HMAC catches it but you lose the whole message.
  • Cover must not already contain U+200B/U+200C. The encoder rejects such covers rather than silently double-counting.

9. When to pick this method

Pick zero-width when the channel preserves Unicode and strips trailing whitespace â€” rendered text in most chat apps, web pages, raw paste boxes. Pick whitespace when the opposite is true (raw files, source code, plain-text email). They fail in opposite conditions, which is why this tool ships both.
