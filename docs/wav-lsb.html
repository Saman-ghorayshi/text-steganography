WAV (16-bit PCM LSB) steganography, explained from zero

The least-significant-bit method for audio. We hide an encrypted secret in the least-significant bit of each 16-bit signed PCM sample. This page walks one tiny payload through every single step, shows you exactly what the sample values look like, and exactly where it breaks.

1. The one-sentence idea

A 16-bit PCM sample is a signed integer in the range −32 768 to +32 767. Changing the lowest bit changes the sample by at most 1 unit out of 65 536 — less than one quantization step, which no human ear can detect. So every sample can quietly carry one payload bit, and the audio still sounds identical.

2. The cast of characters

ThingWhat it isRole in the trick
secretthe message you actually want to sendencrypted first, then hidden
passworda string only you and the recipient knowturns the secret into unreadable ciphertext
coveran innocent-looking WAV file (16-bit PCM)camouflage; payload is written into its sample LSBs
stego WAVthe cover with the payload stored in sample LSBswhat you actually send

LSBleast-significant bit, the rightmost binary digitwhat we flip; we don't touch the other 15 bits of any sample

3. The full pipeline (end to end)

  1. Encrypt. The secret's UTF-8 bytes go through Fernet (AES-128-CBC + HMAC-SHA256) keyed by SHA-256(password). Out comes ciphertext — random-looking bytes. Even if someone later finds the hidden bits, they still need the password.
  2. Length-prefix. Prepend the ciphertext length as a 4-byte big-endian integer. This is the header that lets the decoder know when to stop reading sample LSBs.
  3. Bits. Convert [4-byte length] + ciphertext to one long binary string. Every byte becomes 8 bits, MSB first ({byte:08b}).
  4. Walk samples. Read the cover WAV, unpack the raw audio frames as 16-bit unsigned values (struct.unpack_from("<H", ...)). For each payload bit in order, take the next sample, clear its LSB with u & 0xFFFE, then set it to the payload bit with a bitwise OR.
  5. Rebuild WAV. Write the modified samples back into a new WAV with identical parameters (channels, sample width, sample rate). Any other format would re-quantize and destroy the LSB payload — see §6.
  6. Send. The receiver gets a WAV that sounds identical. They do not need any special software beyond a player that reads WAV bytes exactly.
  7. Decode. Read samples in order, extract every LSB into one long bit string.
  8. Read header. First 32 bits = length in bytes. Read that many more bytes as the ciphertext.
  9. Decrypt. Feed ciphertext + password back through Fernet. Out comes the original secret. Wrong password → Fernet's HMAC rejects it, you get nothing.

4. The worked example, on paper

Let's hide the 1-byte secret 'A' (= 0x41 = 01000001) in a short 16-bit mono WAV cover using password "pw". We'll do every step by hand so you can see the bits. (In practice the cover needs at least 40 samples for this payload; we'll show the first few.)

Step A — encrypt

hide_wav(b"A", "pw", cover_wav) asks Fernet for ciphertext. Fernet adds a random IV, so the real bytes are unpredictable; for this walkthrough pretend the ciphertext comes out to exactly one byte: 

ciphertext = b"\x41"      # 1 byte, value 0100 0001

In real use Fernet's output is much longer (it prepends a version byte, timestamp, IV, and appends an HMAC). The math is identical — just more bits.

Step B — length-prefix

The length of the ciphertext is 1. Encode 1 as a 4-byte big-endian integer, then concatenate the ciphertext:

length (4 bytes):  00 00 00 01
ciphertext (1 byte): 41
payload (5 bytes):  00 00 00 01 41

Step C — to bits

Each byte → 8 bits, MSB first. 5 bytes → 40 bits total. 40 bits need 40 samples, so our cover WAV needs at least 40 PCM samples.

00       00       00       01       41
00000000 00000000 00000000 00000001 01000001

So the full bit string we need to hide is:

00000000 00000000 00000000 00000001 01000001
└────── header (32 bits) ────────┘└ payload ┘

Step D — write into sample LSBs

Assume the first few samples of the cover (in frame order, channel-by-channel) look like this:

sample  0:  +3200   binary: 01111101 00010000
sample  1:  −1582   binary: 11111001 11100010
sample  2:  +8742   binary: 00100010 01000110
sample  3:  −423    binary: 11111110 01101001
sample  4:  +19999  binary: 01001110 00011111
...

We walk samples in order — sample 0 gets bit 0, sample 1 gets bit 1, and so on. For each sample we read it as a 16-bit unsigned value, clear the LSB, then OR in the payload bit:

sample  0:  u = unpack("<H", bytes) = 3200
            binary: 0111110100010000
            clear LSB: 3200 & 0xFFFE = 3200  (was already 0)
            bit 0 = 0  →  3200 | 0 = 3200
            binary: 0111110100010000  (no change)

sample  1:  u = −1582 → packed as unsigned 63954
            binary: 1111100111100010
            clear LSB: 63954 & 0xFFFE = 63954  (was already 0)
            bit 1 = 0  →  63954 | 0 = 63954
            binary: 1111100111100010  (no change)

sample  2:  u = 8742
            binary: 0010001001000110
            clear LSB: 8742 & 0xFFFE = 8742  (was already 0)
            bit 2 = 0  →  8742 | 0 = 8742  (no change)
...

Skip ahead to the interesting bits. The header's last byte is 00000001, so bit 31 (the last header bit) is 1. The payload byte 0x41 = 01000001 gives us bits 32–39:

sample 31:  u = +11203   binary: 00101011 11010011
            bit 31 = 1  →  clear LSB → 11202  →  11202 | 1 = 11203
            binary: 0010101111010011  (LSB was already 1, no change)

sample 32:  u = −983     unsigned: 64553   binary: 11111100 10001001
            bit 32 = 0  →  64553 & 0xFFFE = 64552  →  | 0 = 64552
            binary: 11111100 10001000  (LSB cleared)

sample 33:  u = +7421    binary: 00011101 00000101
            bit 33 = 1  →  7421 & 0xFFFE = 7420  →  | 1 = 7421
            binary: 00011101 00000101  (LSB already 1, no change)

sample 34:  u = −20450   unsigned: 45086   binary: 10101111 11101110
            bit 34 = 0  →  45086 & 0xFFFE = 45086  →  | 0 = 45086
            (no change)

sample 35:  u = +16      binary: 00000000 00010000
            bit 35 = 0  →  16 & 0xFFFE = 16  →  | 0 = 16
            (no change)

sample 36:  u = +44095   binary: 10101100 01101111
            bit 36 = 0  →  44095 & 0xFFFE = 44094  →  | 0 = 44094
            binary: 10101100 01101110  (LSB cleared)

sample 37:  u = +402     binary: 00000001 10010010
            bit 37 = 0  →  402 & 0xFFFE = 402  →  | 0 = 402
            (no change)

sample 38:  u = +999     binary: 00000011 11100111
            bit 38 = 0  →  999 & 0xFFFE = 998  →  | 0 = 998
            binary: 00000011 11100110  (LSB cleared)

sample 39:  u = −3302    unsigned: 62234   binary: 11110011 11011010
            bit 39 = 1  →  62234 & 0xFFFE = 62234  →  | 1 = 62235
            binary: 11110011 11011011  (LSB set)

Notice the giveaway: a natural audio recording has LSBs that are roughly 50/50 ones and zeros — they're mostly noise. Our modified cover has its first ~32 LSBs almost entirely 0 because the length header is small. A steganalysis tool can detect this: see §7.

Step E — decode (reverse)

Read samples in order, extract each LSB. You get the same 40-bit string back:

00000000 00000000 00000000 00000001 01000001

First 32 bits → int(...,2) = 1, so the decoder knows to read 1 more byte. The next 8 bits → 01000001 = 0x41 = b"A". Hand b"\x41" and "pw" to Fernet → original secret.

5. Why only 16-bit PCM?

8-bit WAV (unsigned). 8-bit PCM samples are unsigned (0–255). The LSB math differs because Python's struct.pack/unpack treats them differently, and 8-bit audio has far fewer amplitude levels — flipping the LSB changes the value by 1/256 instead of 1/65 536, which is 256× more audible. The CLI refuses 8-bit covers with sampwidth != 2.

24-bit or 32-bit float WAV. These use different sample widths and bit representations. 24-bit samples need 3-byte unpack; 32-bit float samples store amplitude as IEEE 754 floats where the LSB is a sub-bit of the mantissa. The bit-twiddling (& 0xFFFE | bit) would corrupt the value unpredictably. Different bitcodecs needed, not supported.

Compressed WAVs (ADPCM, A-law, μ-law). These are not PCM — the stored bytes are codec-encoded deltas, not raw amplitude values. Writing into their LSBs would corrupt the codec stream, producing audible glitches or silence. The wave stdlib module can read some of these, but the sampwidth check catches them.

6. Why not MP3 / AAC / OGG?

Lossy codecs destroy LSBs. MP3, AAC, and OGG Vorbis are lossy — they decompose the waveform into frequency bands (MDCT or filter banks), quantize the coefficients, then reconstruct. The reconstructed samples are different from the originals: every sample's LSB is re-quantized and effectively randomized. Your carefully hidden payload bits are gone — silently, with no error signal.

This is why the CLI refuses non-WAV output. The _wav_only() guard checks the output extension and raises ValueError if you try -o stego.mp3. The underlying reason is physics: you cannot hide information in a frequency-band codec that was specifically designed to throw away inaudible detail. For audio steganography, the cover must be lossless, and right now that means uncompressed 16-bit PCM WAV.

FLAC and ALAC are lossless compressed formats that would preserve the LSBs, but Python's stdlib wave module doesn't read them. Supporting FLAC is a future enhancement.

7. Detection

Steganography hides content, not existence. Two things give you away:

Sample-LSB distribution

A natural recording's LSBs look like random noise — roughly 50% ones, 50% zeros. After embedding a small payload, the first N samples have their LSBs set to the length header (which is mostly zeros for typical payloads). A histogram of the first few hundred LSBs will show a suspicious skew toward 0.

Readable length header, no password needed

The decoder reads the first 32 sample LSBs as a 4-byte big-endian integer. If that integer is a plausible payload size (non-zero, fits within the WAV's total sample count), the file almost certainly contains a hidden message — random LSB noise would not produce such a tight prefix. This is the same signal the detect helper uses: 

python steg.py detect -m wav -i stego.wav
[wav] 882000 sample LSB bits
[wav] declared 1 bytes payload that fits the WAV capacity
      (110 246 sample-LSB bytes) — suspicious

Chi-square steganalyzer. A more rigorous approach is a chi-square test on the sample-LSB frequency: split the first N LSBs into blocks, compute the deviation from a 50/50 distribution, and flag if it's statistically significant. Our detect helper uses the simplest honest signal — a readable length header that fits inside the WAV's capacity — but a real steganalysis portfolio uses chi-square as the gold standard.

8. Doing it yourself

  1. Use the demo tab (audio mode) or the CLI: python steg.py encode -m wav -p pw -s "Hello" -c cover.wav -o stego.wav then python steg.py decode -m wav -p pw -i stego.wav.
  2. Pick a cover WAV that is at least a few seconds of 16-bit PCM — capacity is frames × channels / 8 − 4 bytes. A 10-second 44.1 kHz mono WAV holds 441 000 / 8 − 4 ≈ 54 KiB.
  3. The encoder refuses non-WAV output (MP3, AAC, etc.) because lossy re-encode would silently destroy the LSB payload.
  4. To detect: python steg.py detect -m wav -i stego.wav. No password needed — it reads the sample LSB stream directly.

For a programmatic look the Python library is the canonical source: from steg import hide_wav, reveal_wav — takes WAV bytes, returns WAV bytes / secret bytes.

9. Limits (short version)

  • Capacity. 1 bit per sample. A 10-second 44.1 kHz mono WAV hides 441 000 / 8 − 4 ≈ 54 KiB. Stereo doubles it. Plenty for text secrets; tiny for media.
  • Lossy transport is deadly. Any codec that re-quantizes (MP3, AAC, OGG) destroys the payload — silently. Stick to WAV, or a lossless codec that Python can read.
  • Steganalysis is well-studied. LSB in the temporal domain is the textbook steg method and the textbook steganalyzer target. A chi-square estimator will flag a payload-bearing WAV — pick a different cover if you need genuine secrecy.
  • No integrity without the password. A sample flip in transit corrupts the ciphertext; Fernet's HMAC catches it but you lose the whole message — no partial recovery.
  • 16-bit PCM only. 8-bit, 24-bit, 32-bit float, and compressed WAVs (ADPCM, A-law) are refused because their sample representations are incompatible with the & 0xFFFE | bit LSB write.

10. When to pick this method

Pick the audio method when your cover is a WAV file and the transport channel preserves bytes — e.g. sending a WAV as a file attachment, hosting it on a site that doesn't re-encode. Audio covers have huge capacity (a one-minute 44.1 kHz stereo WAV holds ~630 KiB) compared to image or text methods. For text-only channels, whitespace or zero-width work without inventing an audio file. For images, image LSB is the analog — same idea, different cover medium.
